How It Works

Phase 1 — Document Registration
1 Upload original academic document
2 SHA-256 hash is computed (64-char fingerprint)
3 Institution signs hash with RSA-2048 private key
4 HMAC tag generated for integrity
5 (Optional) File encrypted with AES-256-CBC
6 Record mined into Blockchain (Proof of Work)
Phase 2 — Document Verification
1 Verifier uploads document to check
2 SHA-256 hash re-computed from uploaded file
3 Hash searched across entire blockchain
4 RSA signature verified with public key
5 HMAC integrity re-checked
6 Result: Original ✔ or Tampered ✗

Concepts Implemented

SHA-256 Hashing

One-way function producing a 256-bit fingerprint. Avalanche effect — 1 bit change → 50% of hash changes.

RSA Digital Signatures

2048-bit asymmetric keys. Private key signs; public key verifies. Ensures authenticity & non-repudiation.

AES-256-CBC Encryption

256-bit symmetric encryption. Random IV per operation. PKCS#7 padding. Protects stored documents.

HMAC-SHA256

Keyed hash for message authentication. Prevents forgery even when hash algorithm is known.

Blockchain

Each block links to previous via hash. Changing one block breaks all subsequent hashes — immutable ledger.

Proof of Work

Mining finds a nonce so block hash starts with N zeros. Makes tampering computationally expensive.

Merkle Tree

Binary hash tree of transactions. Root stored in block header — tamper-proof transaction summary.

Chain Validation

Full chain re-validated on demand — detects hash corruption, previous-hash mismatch, or PoW violation.

Base64 Encoding

Binary-safe transport for signatures, ciphertext, and IVs. Ensures JSON compatibility of cryptographic data.
